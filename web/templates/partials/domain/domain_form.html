Caddy file_server

Serve files from disk at a URL prefix. Optional—only if you want Caddy to read files directly instead of proxying everything to the app.

How this works

  • Who reads files: the path is resolved on the Caddy (ingress) container, not inside your app process.
  • Add one route: choose Static files (e.g. after collectstatic) or User media (uploads). Static and media must use different public URL prefixes.
  • Mounts: that filesystem path must exist in the Caddy container—usually by bind-mounting the same host folder to the same absolute path on both Caddy and the app (e.g. MEDIA_ROOT and this field both /app/media).
  • URL prefix: must match what the browser requests (STATIC_URL / MEDIA_URL). Caddy strips the prefix and maps the rest under your filesystem root.
  • After saving: redeploy so generated compose labels apply.
Static files

Must match how browsers load assets (leading /). Not the site root alone.

Must exist on the Caddy container. Often same as STATIC_ROOT, with the host volume mounted at this path on Caddy and the app.

User media

Must match MEDIA_URL (any depth, e.g. /media or /app/media).

Must exist on the Caddy container. Usually same absolute path as MEDIA_ROOT, with the same host directory mounted into Caddy and the app.
